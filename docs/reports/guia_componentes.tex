\section{Objetivo de esta guía}
Esta guía explica el subsistema de workers por partes, con un enfoque práctico. Para cada componente se documenta:
\begin{itemize}
\item qué hace;
\item en qué archivo vive;
\item qué recibe;
\item qué entrega;
\item cómo se conecta con los demás.
\end{itemize}

La intención es que esta guía sirva para estudiar el proyecto, exponerlo o retomarlo después sin reconstruir el contexto desde cero.

\section{Vista general del sistema}
La arquitectura final es:

\begin{center}
\textbf{Cliente/Web/App $\rightarrow$ Servidor principal Java $\rightarrow$ Workers Python}
\end{center}

El servidor principal Java se encarga de:
\begin{itemize}
\item autenticación;
\item base de datos;
\item lógica de negocio;
\item coordinación global del sistema.
\end{itemize}

Este repositorio se concentra en el plano de ejecución:
\begin{itemize}
\item tres workers;
\item interfaces gRPC para negocio y control;
\item almacenamiento compartido opcional;
\item métricas, healthchecks y seguridad.
\end{itemize}

\subsection{Resumen del flujo extremo a extremo}
\begin{enumerate}
\item El servidor principal Java decide que debe procesarse una imagen.
\item El servidor principal se conecta a uno de los workers por gRPC.
\item El worker valida la solicitud y la transforma en una tarea interna.
\item La tarea entra a la cola local con prioridad.
\item El scheduler la despacha cuando hay recursos suficientes.
\item El pipeline procesa la imagen.
\item El worker devuelve el resultado por la RPC o, si se configuró un endpoint externo, envía callbacks de progreso, resultado y heartbeat.
\end{enumerate}

\section{Despliegue}

\subsection{Docker Compose}
\textbf{Archivo principal}: \texttt{docker-compose.yml}

\textbf{Qué hace}
\begin{itemize}
\item levanta MinIO como storage compartido;
\item levanta \texttt{worker1}, \texttt{worker2} y \texttt{worker3};
\item levanta Prometheus y Grafana;
\item inyecta certificados y secretos.
\end{itemize}

\textbf{Qué recibe}
\begin{itemize}
\item imágenes Docker;
\item secretos montados desde \texttt{.secrets/};
\item configuración desde variables de entorno.
\end{itemize}

\textbf{Qué entrega}
\begin{itemize}
\item un clúster local reproducible de workers;
\item puertos públicos para gRPC, salud, métricas y consola MinIO.
\end{itemize}

\textbf{Cómo se conecta}
\begin{itemize}
\item MinIO se usa para storage compartido;
\item los workers exponen gRPC por mTLS;
\item Prometheus scrapea las métricas de cada worker;
\item Grafana consume Prometheus.
\end{itemize}

\section{Worker}

\subsection{Visión general}
Cada worker es un ejecutor especializado. Su responsabilidad es aceptar tareas, administrarlas en una cola priorizada local, decidir cuándo puede correrlas y ejecutar el pipeline de imagen.

\subsection{Configuración del worker}
\textbf{Archivo principal}: \texttt{worker/config.py}

\textbf{Qué hace}
\begin{itemize}
\item carga puertos, límites, storage, TLS, tracing, OCR e inferencia;
\item define si el worker reporta callbacks a un servidor principal externo;
\item define límites duros de entrada.
\end{itemize}

\textbf{Entradas principales}
\begin{itemize}
\item \texttt{WORKER\_BIND\_*};
\item \texttt{WORKER\_COORDINATOR\_TARGET};
\item \texttt{WORKER\_MAX\_REQUEST\_BYTES};
\item \texttt{WORKER\_MAX\_BATCH\_SIZE};
\item \texttt{WORKER\_MAX\_IMAGE\_*};
\item \texttt{WORKER\_STORAGE\_*}.
\end{itemize}

\subsection{Orquestador interno}
\textbf{Archivo principal}: \texttt{worker/core/node.py}

\textbf{Qué hace}
\begin{itemize}
\item mantiene la cola local;
\item lanza el scheduler;
\item maneja admisión;
\item registra tareas activas y completadas;
\item procesa cancelaciones;
\item coordina retries;
\item actualiza salud, métricas y reporter;
\item restaura pendientes y resultados al reiniciar.
\end{itemize}

\textbf{Entradas}
\begin{itemize}
\item tareas nuevas desde gRPC;
\item snapshots de recursos;
\item resultados del execution manager;
\item eventos de cancelación y shutdown.
\end{itemize}

\textbf{Salidas}
\begin{itemize}
\item decisiones de aceptación o rechazo;
\item persistencia de estado local o compartido;
\item callbacks al servidor principal si existe endpoint configurado.
\end{itemize}

\subsection{API de negocio del worker}
\textbf{Archivo principal}: \texttt{worker/core/business\_api.py}

\textbf{Qué hace}
\begin{itemize}
\item traduce solicitudes del contrato \texttt{ImageNodeService} a tareas internas;
\item construye \texttt{Task} a partir de filtros y payload;
\item valida tamaño, batch, cantidad de filtros, dimensiones y número de píxeles;
\item espera el resultado y lo devuelve como bytes o como path.
\end{itemize}

\subsection{Parser de filtros}
\textbf{Archivo principal}: \texttt{worker/core/filter\_parser.py}

\textbf{Qué hace}
\begin{itemize}
\item interpreta filtros de texto;
\item convierte cada filtro en una \texttt{TransformationSpec};
\item detecta el formato de salida.
\end{itemize}

\textbf{Ejemplos}
\begin{itemize}
\item \texttt{resize:640x480}
\item \texttt{rotate:90}
\item \texttt{watermark:Demo|10|10|white}
\item \texttt{format:webp}
\item \texttt{ocr}
\item \texttt{inference}
\end{itemize}

\section{Scheduler local}

\subsection{Cola priorizada}
\textbf{Archivo principal}: \texttt{worker/scheduler/priority\_queue.py}

\textbf{Qué hace}
\begin{itemize}
\item encola tareas;
\item permite sacarlas en orden de prioridad efectiva;
\item soporta remoción y espera asíncrona.
\end{itemize}

\subsection{Scoring}
\textbf{Archivo principal}: \texttt{worker/scheduler/scoring.py}

\textbf{Qué hace}
\begin{itemize}
\item calcula el score de cada tarea;
\item mezcla prioridad, aging, deadline, costo y tamaño.
\end{itemize}

\textbf{Idea del score}
\begin{itemize}
\item las tareas urgentes suben;
\item las viejas ganan aging;
\item las caras o grandes bajan;
\item los retries penalizan;
\item se evita starvation.
\end{itemize}

\subsection{Costo estimado}
\textbf{Archivo principal}: \texttt{worker/scheduler/cost\_model.py}

\textbf{Qué hace}
\begin{itemize}
\item estima el tiempo de servicio;
\item aprende de ejecuciones pasadas usando promedio exponencial.
\end{itemize}

\section{Ejecución y procesamiento de imágenes}

\subsection{Gestión de recursos}
\textbf{Archivo principal}: \texttt{worker/execution/resource\_manager.py}

\textbf{Qué hace}
\begin{itemize}
\item estima requerimientos de una tarea;
\item toma snapshots de CPU, RAM y ocupación;
\item decide si una tarea puede correr ahora.
\end{itemize}

\subsection{Execution Manager}
\textbf{Archivo principal}: \texttt{worker/execution/execution\_manager.py}

\textbf{Qué hace}
\begin{itemize}
\item arranca la ejecución real;
\item limita concurrencia con semáforos;
\item decide entre thread y proceso dedicado;
\item maneja cancelación;
\item limpia salidas parciales.
\end{itemize}

\subsection{Procesador de imágenes}
\textbf{Archivo principal}: \texttt{worker/execution/image\_processor.py}

\textbf{Qué hace}
\begin{itemize}
\item carga la imagen de entrada;
\item aplica transformaciones;
\item ejecuta OCR o inferencia si la tarea lo pide;
\item serializa el formato final;
\item persiste el resultado.
\end{itemize}

\textbf{Operaciones integradas}
\begin{itemize}
\item grayscale;
\item resize;
\item crop;
\item rotate;
\item flip;
\item blur;
\item sharpen;
\item brightness/contrast;
\item watermark/text;
\item conversión de formato.
\end{itemize}

\textbf{Formatos soportados}
\begin{itemize}
\item JPG
\item PNG
\item TIF
\item WEBP
\item BMP
\item GIF estático
\item ICO
\end{itemize}

\textbf{Punto importante}
Las operaciones costosas integradas fueron implementadas de manera cooperativa, para que la cancelación normal no dependa de matar procesos en la mayoría de los casos.

\section{Persistencia y storage}

\subsection{Storage abstracto}
\textbf{Archivo principal}: \texttt{worker/core/storage.py}

\textbf{Qué hace}
\begin{itemize}
\item unifica acceso a distintos backends;
\item lee, escribe, lista y borra URIs;
\item soporta local, \texttt{file://}, \texttt{http(s)} y \texttt{s3://}.
\end{itemize}

\subsection{Persistencia del worker}
\textbf{Archivo principal}: \texttt{worker/core/state\_store.py}

\textbf{Qué hace}
\begin{itemize}
\item guarda resultados terminales;
\item guarda tareas pendientes;
\item guarda eventos de progreso y resultado mientras esperan \texttt{ack}.
\end{itemize}

\textbf{Subcomponentes}
\begin{itemize}
\item \texttt{StateStore}: cache durable de completados;
\item \texttt{PendingTaskStore}: tareas en cola o retry;
\item \texttt{EventSpoolStore}: cola durable de eventos hacia el servidor principal.
\end{itemize}

\section{gRPC}

\subsection{Contrato de negocio}
\textbf{Archivo principal}: \texttt{proto/imagenode.proto}

\textbf{Qué hace}
\begin{itemize}
\item define la interfaz de negocio para pedir procesamiento;
\item expone modos unary, streaming y batch.
\end{itemize}

\textbf{Métodos principales}
\begin{itemize}
\item \texttt{ProcessToPath}
\item \texttt{ProcessToData}
\item \texttt{UploadLargeImage}
\item \texttt{StreamBatchProcess}
\item \texttt{ProcessBatch}
\item \texttt{HealthCheck}
\item \texttt{GetMetrics}
\end{itemize}

\subsection{Contrato interno de control}
\textbf{Archivo principal}: \texttt{proto/worker\_node.proto}

\textbf{Qué hace}
\begin{itemize}
\item define el control fino del worker;
\item define el endpoint opcional de callbacks.
\end{itemize}

\textbf{Servicios}
\begin{itemize}
\item \texttt{WorkerControlService}
\item \texttt{CoordinatorCallbackService}
\end{itemize}

\textbf{Nota}
El nombre \texttt{CoordinatorCallbackService} se conserva por compatibilidad de proto, pero en la arquitectura final representa el endpoint de callbacks del servidor principal Java.

\subsection{Servidor de control del worker}
\textbf{Archivo principal}: \texttt{worker/grpc/servicer.py}

\textbf{Qué hace}
\begin{itemize}
\item implementa \texttt{SubmitTask};
\item implementa \texttt{GetNodeStatus};
\item implementa \texttt{CancelTask};
\item implementa \texttt{DrainNode};
\item implementa \texttt{ShutdownNode}.
\end{itemize}

\subsection{Servidor de negocio del worker}
\textbf{Archivo principal}: \texttt{worker/grpc/business\_servicer.py}

\textbf{Qué hace}
\begin{itemize}
\item expone \texttt{ImageNodeService};
\item usa internamente \texttt{business\_api.py};
\item reutiliza el mismo motor del worker.
\end{itemize}

\subsection{Reporter}
\textbf{Archivo principal}: \texttt{worker/grpc/reporter.py}

\textbf{Qué hace}
\begin{itemize}
\item mantiene el canal de callbacks si existe un endpoint externo configurado;
\item envía \texttt{ProgressEvent};
\item envía \texttt{ExecutionResult};
\item envía \texttt{Heartbeat};
\item reintenta con backoff si el endpoint no responde.
\end{itemize}

\textbf{Punto importante}
Si no se configuró \texttt{WORKER\_COORDINATOR\_TARGET}, el reporter queda desactivado y el worker sigue operando.

\subsection{Seguridad gRPC}
\textbf{Archivo principal}: \texttt{worker/grpc/security.py}

\textbf{Qué hace}
\begin{itemize}
\item construye credenciales TLS del servidor;
\item construye credenciales TLS del cliente;
\item soporta mTLS en ambos sentidos.
\end{itemize}

\section{Cómo seguir una solicitud real}

\subsection{Camino normal}
\begin{enumerate}
\item El servidor principal Java invoca \texttt{ImageNodeService} o \texttt{WorkerControlService}.
\item El worker recibe la request.
\item La request se valida y se convierte en una \texttt{Task}.
\item La tarea entra a la cola local.
\item El scheduler la saca cuando hay recursos.
\item El execution manager corre el pipeline.
\item El resultado se persiste.
\item El worker responde por la RPC o envía callbacks si el endpoint externo está configurado.
\end{enumerate}

\subsection{Camino con callbacks}
\begin{enumerate}
\item El servidor principal expone un endpoint compatible con \texttt{CoordinatorCallbackService}.
\item El worker envía \texttt{Heartbeat}, \texttt{ReportProgress} y \texttt{ReportResult}.
\item Si el endpoint no responde, los eventos quedan en spool durable.
\item Cuando el canal vuelve, los eventos se reenvían.
\end{enumerate}

\textbf{Componente} & \textbf{Decisión principal} \\

\bottomrule
\end{longtable}

\section{Puntos clave para explicar el proyecto}
Si necesitas resumir el sistema oralmente, estas son las ideas más importantes:
\begin{itemize}
\item el servidor principal Java coordina y los workers ejecutan;
\item este repo no maneja auth ni BD de negocio;
\item cada worker tiene cola local con prioridad, backpressure y persistencia;
\item el sistema usa gRPC, mTLS, MinIO, Prometheus y Grafana;
\item el worker soporta OCR e inferencia reales mediante adaptadores.
\end{itemize}

\section{Conclusión}
El proyecto quedó dividido de manera clara entre lógica de negocio y plano de ejecución. El servidor principal Java conserva autenticación, persistencia corporativa y coordinación global. Este repositorio aporta un subsistema de workers Python capaz de priorizar, admitir, ejecutar, persistir y reportar tareas de procesamiento de imágenes. Con esta separación, la solución es entendible, demostrable y extensible sin mezclar responsabilidades.
